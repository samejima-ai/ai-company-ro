# 機能仕様 — ヤジ（大喜利MCボット）

> SSOT は `docs/HANDOFF.md`。本 SPEC は HANDOFF を「機能×条件」の粒度に落とした実装契約。
> 対象は **MVP Phase 0（無料運用・課金導線なし）**。投げ銭関連は「後で載る前提で設計し、Phase 0 では実装しない」。

## WHY（目的）

テキスト対話で投げ銭を稼ぐ大喜利MCボットの「脳」と「人間ゲート付き改善ループ」を、最小コストで検証する。
Phase 0 の唯一の関門は **7日リテンション**。これが立つまで課金は付けない。
稼ぎ方の原則は操作（manipulation）ではなく「一貫した魅力と芸」。依存搾取・色恋課金・射幸性は構造的に排除する（→ DONT.md）。

## WHAT（機能定義）

### F1 Discord I/O {#f1-discord-io}

#### 概要
1サーバー上で、テキストメッセージを受信し、ヤジとして逐次応答する。

#### 条件
- 接続: 単一 Discord bot として 1 ギルドに常駐する。`DISCORD_TOKEN` は `.env` から読む。実値はコミットしない。
- 受信対象: bot 自身の発言・他 bot の発言は処理しない（無限ループ防止）。
- 応答トリガー: メンション、リプライ、または「大喜利お題チャンネル」として設定された channel ID（`OOGIRI_CHANNEL_IDS`）での発言。
- 応答遅延: 受信から応答開始まで p95 で 30 秒以内。長文生成中は typing インジケータを出す。
- 多重処理: 同一ユーザーの連投はユーザー単位で直列化し、古い未処理メッセージは最新で上書きしない（全て順次処理する）。
- 失敗時: モデル呼び出し・送信が失敗した場合、ユーザーには 1 度だけキャラのままの軽い詫び（例「いま頭ショートした。もう一回投げて」）を返し、詳細はログに残す。例外でプロセスを落とさない。

#### Priority
critical

#### UX制約
- Must 閾値: 応答開始まで p95 30 秒以内（NFR の P/A と連動）。
- 禁止挙動: 無言で落ちる／同一エラーを連投する／@everyone 等の broadcast。
- 参考類似サービス: Neuro-sama（リアルタイム掛け合いのテンポ感）。

#### エッジケース
- 空メッセージ・添付のみ・スタンプのみ: お題として成立しない旨をキャラのまま 1 行で返す。
- 極端な長文（>2000 字）: 先頭規定長のみをお題として扱う。

### F2 対話エンジン {#f2-対話エンジン}

#### 概要
モデルルーター＋ヤジ system prompt（`personas/riko_system_prompt.md`）＋関連記憶を束ねて応答テキストを生成する。

#### 条件
- ルーティング: 通常会話は高速・小型モデル、難所（高額投げ銭の見せ場・複雑な大喜利・安全境界の判断）だけ大型モデルへエスカレーションする。プロバイダは非固定（`ModelRouter` インターフェース経由）。具体モデルは決定時にベンチマークし REGIME.md / .env で差し替え可能にする。
- persona 固定: system prompt は `personas/riko_system_prompt.md` を単一の正本として読み込む。コード内に persona 文字列を二重定義しない。
- 記憶注入: 毎ターン、当該ユーザーの関連記憶のみを context に注入する（全件注入禁止）。注入対象は呼び名・鉄板ネタ・（Phase 1 以降）課金履歴。
- 記憶の用途制限: 「打ち明け話（悩み・健康・個人事情）」フラグ付き記憶は、いじり・営業の材料に使わない（プロンプト組み立て時に除外する）。
- 出力長: 1 応答は Discord 1 メッセージ（規定上限内）に収める。超える場合は要約して 1 通にする。

#### Priority
critical

### F3 記憶層 {#f3-記憶層}

#### 概要
ユーザー別の長期記憶を保持し、一貫性（同じ相手に同じ世界観で接する）の土台を作る。

#### 条件
- ストア: MVP は SQLite（KV 中心）。スキーマは `spec/domain.ts` を正本とする。育てばベクタ検索へ移行可能な抽象（`MemoryStore` インターフェース）に閉じる。
- 単位: Discord user ID をキーに、`UserProfile`（呼び名・初回観測日・最終観測日）と `MemoryEntry[]`（kind 付き）を保持する。
- kind 区別: 記憶は `nickname` / `running_gag`（鉄板ネタ）/ `confession`（打ち明け話）/ `tip_history`（課金履歴・Phase 1）に分類する。`confession` は F2 の注入で営業・いじりに使わない。
- 上限: 1 ユーザーあたり `MemoryEntry` は既定 200 件まで。超過時は最古の非 `confession` から削除する（`confession` は保持優先）。
- 削除要求: ユーザーが自分の記憶削除を求めた場合、当該 user の全 entry を物理削除できる（個人情報保護）。

#### Priority
critical

### F4 モデレーション {#f4-モデレーション}

#### 概要
入口（受信時）と出口（送信前）の両方で危険・NG を弾く二重フィルタ。

#### 条件
- 入口: プロンプトインジェクション、未成年関与の疑い、NGトピック（→ DONT.md / §5.9）を検出する。検出時はモデルに渡さず、キャラのままかわす定型（説教しない）を返す。
- 出口: 生成テキストがペルソナ逸脱・有害発話・NG（属性攻撃・性的・依存助長・断定的な医療/法律/金融助言・個人情報暴露・恋愛錯覚での課金誘導）に当たらないか検査する。当たれば再生成を 1 回試み、なお当たれば安全な定型に差し替える。
- りりマニュアル由来 red-team: 「偽の悩み設定への同調」「『助けたい』を引き出す誘導」「やみえい（病み営業）」「未来形色恋」「連投課金の煽り」を**ボット自身が行わない**ことを出口で検査する（検査観点は sensors/inferential.md と spec/invariants.feature の Evil Path）。
- フェイルセーフ: モデレーション機構自体が失敗した場合は「通さない」側に倒す（送信を中止し定型に差し替え、ログに残す）。

#### Priority
critical

### F5 ログ {#f5-ログ}

#### 概要
改善ループの入力として、全やり取りを保存する。

#### 条件
- 保存対象: 受信メッセージ、注入記憶の参照、ルーティング結果（使用モデル）、生成テキスト、モデレーション判定、応答レイテンシ、エラー。
- 保存先: SQLite の `interaction_log`（スキーマは `spec/domain.ts`）。
- 個人情報: 生ログにユーザー識別子は user ID のみ。氏名・連絡先等は保存しない（収集しない）。
- 保持: ログは改善ループ用。ユーザー削除要求時は当該 user 分を削除できる。

#### Priority
standard

### F6 簡易eval＋改善ループ {#f6-eval改善ループ}

#### 概要
保存ログから自動指標を出し、**人間ゲート**を通してのみ「速い更新」（persona prompt・記憶・NG辞書の追記）を本番反映する。

#### 条件
- 自動指標（eval）: 会話継続数／1人あたり会話数、安全違反検出件数（出口で弾いた数）、応答レイテンシ分布。目的関数は投げ銭単独にしない（Phase 0 は投げ銭なし）。複合指標＝視聴維持＋安全違反ゼロを主に見る。
- 人間ゲート: persona prompt・NG辞書・記憶補正の変更は、人間（経営者）の承認なしに本番反映しない。自己学習による自動改変は禁止（Tay化回避）。
- 速い更新のみ: MVP では fine-tune を封印し、prompt・辞書・記憶の追記だけで回す。
- 反映形式: 変更は PR として提出し、人間がマージして初めて反映される（→ REGIME.md dev_mode=github_assisted）。

#### Priority
standard

## 制約
- 言語/実行: TypeScript（Node.js, ESM）。Discord 接続は discord.js。
- ARC: monolith（単一の常駐 bot プロセス）。NFR の同時接続が増えた場合の移行先は realtime-pubsub（`spec/state-diagrams.md` 注記）。
- 秘密情報: `.env` のみ。実値はコミットしない。`.env.example` に形だけ置く。
- 安全の二重化: NG/安全は CLAUDE.md（確率的遵守）だけに頼らず、F4 のコード側モデレーション＋センサー／CI で機械検査する。
- Phase 境界: 投げ銭・Stripe・最優先キュー・SKU は Phase 1 以降。Phase 0 では実装しない（インターフェースの席だけ用意する）。

## 機能間相互作用（永続化レイヤ・状態共有）

### 共有レイヤ一覧
- SQLite DB（既定ファイル `data/oogiri.db`、パスは env で上書き可）:
  - `user_profile`: F3 が書き、F2 が読む。
  - `memory_entry`: F3 が書き、F2 が読む（`confession` は F2 注入から除外）。
  - `interaction_log`: F5 が書き、F6 が読む。

### 副作用マトリクス
| 機能実行 | 影響先機能 | 副作用内容 |
|---|---|---|
| F3 記憶追記（`confession`） | F2 対話エンジン | 注入対象から除外され、営業・いじりに使われない |
| F3 記憶上限超過 | F3 自身 | 最古の非 `confession` から削除（`confession` は保持優先） |
| ユーザー削除要求 | F3・F5 | 当該 user の `user_profile`/`memory_entry`/`interaction_log` を同一トランザクションで物理削除 |
| F4 出口 FAIL | F5 ログ | 弾いた事実と理由をログに残し、eval の安全指標に計上 |

### 同時刻整合性ルール
- ユーザー削除は、`user_profile`・`memory_entry`・`interaction_log` を同一トランザクション内で削除する（孤立行を残さない）。
- 同一ユーザーのメッセージ処理は user 単位で直列化し、記憶の読み書き競合を避ける。
