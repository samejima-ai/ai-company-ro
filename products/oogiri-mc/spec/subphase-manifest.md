## 選定モード（L0 サブフェーズ）

| サブフェーズ | 判定 | モード | 理由 | 確度 |
|---|---|---|---|---|
| L0-1 業務意図 | 起動済み | — | 必須（SPEC.md 生成済み） | — |
| L0-2 ドメインモデル | 起動 | 完全 | S1 = データ保存あり（ユーザー別記憶・ログ）、SQLite、複雑な構造 | 確定 |
| L0-3 API 契約 | 起動 | 簡易 | S2 = 外部システム（Discord/LLM）に接続するが API は公開しない。内部モジュール境界の関数署名のみ | 確定 |
| L0-4 状態遷移 | 起動 | 簡易 | S3 = 画面なし。ただしリアルタイム処理パイプラインと遅い改善ループにライフサイクルあり。Mermaid のみ | 確定 |
| L0-5 認可 | 起動 | 簡易 | S4 = マルチユーザー＋ロール（user/課金者/経営者）。リソース所有権グラフは MVP では不要。マトリクスのみ | 確定 |
| L0-6 不変条件 | 起動 | — | L0-2/L0-3/L0-4/L0-5 の 4 サブフェーズ起動による自動判定。層間整合＋安全（Evil Path）の検証が必須 | 確定 |

実行順: L0-2 → (L0-3 ‖ L0-4) → L0-5 → L0-6

## 基本 5 問の回答記録（HANDOFF からの読み取り）
- S1（保存/DB）: あり。ユーザー別長期記憶（呼び名・鉄板ネタ・打ち明け話・将来の課金履歴）＋全ログ保存。MVP は SQLite。
- S2（外部連携）: あり。Discord API・LLMプロバイダ。公開 API は持たない（bot は consumer）。
- S3（画面/遷移）: 独自画面なし（Discord ネイティブ）。ただし「リアルタイム経路」と「2速ループの遅い改善」に状態あり。
- S4（マルチユーザー/権限）: あり。一般ユーザー／（Phase 1）課金者／経営者（人間ゲート）のロール。
- S5（時間/承認での自動状態変化）: あり。ログ→自動eval→人間ゲート→更新反映の承認駆動フロー。
